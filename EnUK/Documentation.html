=== HIDDEN-GARDEN ===

Hidden-garden is a secure way to send a message by email, eventually including attachments, without the help of a trusted third party (this is a free tool). By "secure", it is understood that a person in between the exchange of emails with the person you are conversing will will not have the possibility to decrypt neither the content of the confidential message nor any associated attachments, unless of course he/she can deploy considerable resources beyond the reach of the average Joe.
The other advantage of this tool is that it does not require the installation of software, a simple Internet browser is sufficient for both the sender and recipient of the messages.

Motivations

Develop a way to exchange encrypted messages has always been difficult so far (at least for me):
  • One can buy a certificate and install it within its email client but it could be expensive.
  • A dedicated software can be installed to encrypt messages but are we really sure it is secure?
  • One can use an online tool but can those who publish it be trusted?
Hidden-garden meets at least these needs: no "trusted" third party is required, no software is installed, no mystery about its content:the source code is available, everyone can scan it thouroughly. As you may have already guessed, Hidden-garden was originally developed for my own needs as I could not find what I wanted on the net. Furthermore, I am curious about what others will think about it, so feel free to change / amend / transform the tool according to your needs and to send me your comments, suggestions or invitations for a drink. To be honest, this is the first time I participate to an open-source project, I do not pretend this will be the innovation of the year, only a modest contribution to the effort of having the right for confidentiality among friends or colleagues.

How does it work?

Before explaining the operating principle, let me summarize, without technical details, the most common current cryptographic methods:
  • There exist "symmetric key encryption" algorithms where the same password is used to encrypt and to decrypt a message. This is very effective but is the cause of an issue: how to send the password safely if someone else is stalking you ?
  • There also exist "public-key encryption" algorithms where one password is used to encrypt the message (it is called the "public key") but does not allow to decipher it, and a different password is used to decrypt the message (it is called the "private key"). Thus you can send the public key over the Internet without risk since, even if it is intercepted by a malicious third party ("eavesdropping"), it will not allow messages decryption. The downside of this method is that it only works for messages which do not exceed a few dozen characters in size.
Hence, the solution used to encrypt messages safely without limitation in its size is to combine both methods:
  1. The sender generates a pair of public and private keys and sends the public key to the recipient
  2. The recipient generates a symmetric key, encrypts it using the recipient's public key and sends it back
  3. The sender decrypts the symmetric key with its private key. He can now encrypt messages as long as he wishes with the symmetric key and send it to the recipient.
  4. The recipient decrypts the file with the symmetric key that he kept securely.
At no time confidential information was exchanged in clear between the interlocutors. Remember that the public key of the sender does not need to be concealed. The principle stated above is obviously not new. You will easily find many sites where all of this is presented in details. My aim is only to facilitate its implementation.

How to use it?

At first sight, it may look a bit tedious, but once in place it will be easy to use. Follow these steps :
  1. Download Hidden-garden files on Github
  2. Open the "1_MiseEnPlace_FrFR.html" file in a Web browser. If an alert asks you to authorize the execution of a script, accept it.
    If you cannot overcome it, use a least restrictive Internet browser (such as Firefox, for example) or change Internet options so that you can run scripts locally on your workstation.
  3. Click the "File to be sent" button.

    As its name implies, send the file to the person to whom you wish to write in secret. Warn him/her to not throw your email away thinking it hides a virus.
  4. Click the "Private Key to hold" button.

    As its name also suggests, preciously keep this file that will serve you later.
  5. The recipient is on his side run the file you sent. By clicking on a button it will generate a file that contains an encryption key that will send you back and allow you to encrypt messages. It will also generate a file containing a secret code to keep securely (and therefore not to communicate to anyone, obviously).
  6. The sender will now be able to encrypt and send a message. To do this, open the file named "2_ChiffrerUnMessage_FrFR.html". Select your private key and the encryption key previously sent by the recipient. If you put these two files in the same directory as the one named "2_ChiffrerUnMessage_FrFR.html" they are detected automatically and you are not requested to select them, remember it for next time.
    Then enter the message to be encrypted. If you wish to join one or more file attachments (like what you would do with an email), click "Add File" button as many times as you have files to attach.

    Then click the "Next" button to download the result that consists of a single file to send by email to the recipient. Send also the file named "2_DechiffrerUnMessage_FrFR.html" that will allow him/her to decipher your message.
  7. The recipient can then open the file named "2_DechiffrerUnMessage_FrFR.html", select the file containing the secret code that was generated in the previous step and the encrypted message that the sender has sent it, then click "Decrypt" to see view the contents of the decrypted message and the list of attachments if any.

A few more details

  • Hidden-garden was intentionally developed so that everything fits in very few files to facilitate the installation and copy (for example, if you create a directory by the recipient of your messages). Personally I always put off by the great outpourings of associated directories and subdirectories.
  • It's ugly ? Yes, it's true, I'm not a web designer. If you are skilled in the area, treat yourself, I'm interested in any improvement.
  • If you want to translate the tool into another language, there are not yet handy solutions other than reviewing manually the 3 files of the tool. However, to facilitate the task a little, all the labels contained in the scripts (javascript) are gathered in the header.
  • Compatibility with the HTML5 standard has been checked, but there is no guarantee it will work on all browsers and on all platforms.
  • The generation time of an encrypted message can be very long if the attachments are large and numerous. Again, nothing is guaranteed, it must be tested.
  • The generation of private and public keys of the sender can be time and an alert will warn you in the browser. You can generate words shorter passes and therefore faster to produce but the security level achieved will be highly degraded, it is not advisable. If you're interested, search and edit the file parameter values ​​"1_MiseEnPlace_FrFR" lines which are written var sizePass = 512;, then further var randomPwd = randomPassword(12); and finally var kg = generateRsaKeys(unique_id,2048,"10001");
  • The exchange of keys to send messages secrets work in only one direction. The recipient must follow the same implementation process described above if he/she wants to send a response in a secure way. It's a bit tedious, it should be changed in the future to make it easier to use both ways.
  • Hidden-garden does not verify the identity of the person with whom you are in contact. This is an area for future improvement if I find time for developing it.
  • I expect to be reproched to provide another free tool facilitating conversations between people with bad intentions. Without denying it, I still think we have the right to send a letter and ensure nobody reads it before it is delivered. Moreover, Hidden-garden only allows to conceal the content of the message but not the identity of conversants. People involved in illegal business usually hide their accomplices. This is not possible with Hidden-garden.
  • The tool relies openly on other open source developments on the Internet: crypto-js, JSBN library of cryptographic functions, and other pieces of code grabed here and there in different forums. If I forgot to mention a valuable contributor who in turn feels offended, please forward it to me so that I add it to this document.
